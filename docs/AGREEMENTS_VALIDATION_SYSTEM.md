# Agreements Validation System

A production-ready, streaming validation system that enforces repository structure through Git-only introspection with order-independent digest verification.

## Overview

The Agreements Validation System provides:
- **Top-level only validation**: No recursion, only root entries
- **Git-only operations**: Pure Git object inspection via libgit2
- **Order-independent digests**: Commutative SHA-256 sum over names + rules
- **SARIF output**: Standardized results for CI integration
- **Streaming architecture**: Single-pass, constant-memory validation
- **No overrides**: Allow-list only, default reject

## Architecture

### Streaming Actor Flow

1. **Dispatcher** (Intake)
   - Discovers `.hooksmith/agreements/*.json` files
   - Loads and validates JSON schema
   - Enforces invariants (mode=tree, scope=top-level, policy=allow-only)

2. **Researcher** (Single Pass)
   - Opens tree iterator (libgit2)
   - Single walk of the tree, evaluate + hash as we go
   - No buffering/sorting

3. **Mandator** (Streaming)
   - For each entry: Tree → require name ∈ allow_dirs
   - For each entry: Blob → require name ∈ allow_files or matches *.ext
   - On first miss ⇒ fail fast

4. **Auditor** (Streaming Set-Digest)
   - Order-independent digest: for each name, compute h_i = SHA256(name) and sum them as a 256-bit little-endian integer
   - Commutative → no sort, stable across traversals
   - Rules digest once: H_rules = SHA256(canonical_rules_json)
   - Final: SHA256(hex(H_subject) + "\n" + hex(H_rules))

5. **Reporter** (Counters)
   - Increments counters (files/dirs)
   - Keep only a small ring buffer of first N names for SARIF context

6. **Triage Officer** (Orchestration)
   - Emits SARIF
   - Exits non-zero on any error

## Agreement Format

```json
{
  "version": "1",
  "mode": "tree",
  "policy": "allow-only",
  "allow-dirs": [".github", ".hooksmith", "crates", "src", "tests"],
  "allow-files": ["README.md", "Cargo.toml", "*.md"],
  "subject": {
    "scope": "top-level"
  },
  "digest": {
    "algo": "sha256",
    "value": "<computed-digest>"
  }
}
```

## Usage

### Local Validation
```bash
# Validate all agreements against HEAD
cargo run -p hooksmith-validate -- --ref HEAD

# Validate against specific ref
cargo run -p hooksmith-validate -- --ref main --root .
```

### CI Integration
```yaml
- name: Validate agreements
  run: cargo run -p hooksmith-validate -- --ref ${{ github.sha }} --root . > agreements.sarif
```

## Digest Computation

The digest ensures both repository structure and validation rules are tamper-evident:

1. **Subject**: Order-independent sum of SHA256(name) for each root entry
   - Commutative: H_subject = Σ SHA256(name_i) mod 2^256
   - No sorting required, stable across traversals
2. **Rules**: Canonical JSON of mode, policy, allow_dirs, allow_files
3. **Digest**: `SHA256(hex(H_subject) + "\n" + hex(SHA256(rules)))`

## Key Features

- **Streaming**: Single-pass, constant-memory validation
- **Order-independent**: Commutative digest computation
- **Git-only**: No working tree required, pure Git object inspection
- **Top-level only**: No recursion, no path globs beyond simple `*.ext`
- **No overrides**: Allow-list only, default reject
- **Gitignore validation**: Automatic validation of `.gitignore` content via `xtask gen-gitignore --validate`

## Gitignore Integration

The validation system automatically validates that the `.gitignore` file matches the expected content generated by `cargo xtask gen-gitignore --validate`. This ensures:

- **Consistency**: `.gitignore` is always in sync with repository state
- **Deterministic**: Generated content is predictable and version-controlled
- **Validation**: Any drift is caught during agreement validation

### Hierarchical .gitignore Structure

The project uses a hierarchical `.gitignore` structure with specialized ignore files in key directories:

- **Root `.gitignore`**: Global project-wide ignores
- **`crates/.gitignore`**: Rust crate-specific artifacts
- **`docs/.gitignore`**: Documentation build artifacts
- **`scripts/.gitignore`**: Script-specific artifacts
- **`examples/.gitignore`**: Example-specific artifacts
- **`tests/.gitignore`**: Test-specific artifacts
- **`tools/.gitignore`**: Tool-specific artifacts

### Gitignore Validation Process

1. **Discovery**: System checks if `.gitignore` exists in the repository root
2. **Validation**: Runs `cargo xtask gen-gitignore --validate` to compare current vs expected content
3. **Subdirectory Check**: Validates presence of all expected subdirectory `.gitignore` files
4. **Reporting**: Includes gitignore validation results in SARIF output
5. **Failure**: Any mismatch causes validation to fail with detailed error reporting

### Gitignore Generation

The `xtask gen-gitignore` command provides:

```bash
# Generate .gitignore with current content
cargo xtask gen-gitignore

# Validate existing .gitignore
cargo xtask gen-gitignore --validate

# Show what would be generated without writing
cargo xtask gen-gitignore --dry-run

# Generate minimal .gitignore based on actual Git tree state
cargo xtask gen-gitignore --minimal
```

### Minimal Gitignore Generation

The `--minimal` flag generates a `.gitignore` that:

1. **Reads actual Git tree**: Uses `git ls-tree HEAD` to get current state
2. **Existence-driven**: Only includes patterns for files/dirs that actually exist
3. **Eliminates phantom patterns**: No ignore rules for non-existent items
4. **Perfect alignment**: Always matches the actual repository state

The generated `.gitignore` includes:
- Rust build artifacts (`/target/`, `Cargo.lock`)
- IDE and editor files (`.vscode/`, `.idea/`, `*.swp`)
- OS-specific files (`.DS_Store`, `Thumbs.db`)
- Logs and temporary files (`*.log`, `*.jsonl`, `/logs/`)
- Generated files (completions, WIT files, etc.)
- Test artifacts and demos
- Git worktree helpers
- Rust helper binaries
- **SARIF output**: Standardized results for GitHub Code Scanning
- **Fail-fast**: CI stops on first error
- **Tamper-evident**: Digest changes if structure or rules change

## Acceptance Criteria

✅ **Only top-level names considered** (no `/` in evaluated names)  
✅ **New root files/dirs fail CI** unless added to allow_* AND digest rotated  
✅ **Deterministic digest** reproduces across machines  
✅ **Git-only operations** via libgit2  
✅ **SARIF 2.1.0 output** for CI integration  
✅ **Five-actor architecture** with clear separation of concerns
